================================================================================
                    AGROMARKET: A BEGINNER'S GUIDE
================================================================================

Welcome to the Agromarket project! This document will explain how the entire 
project works from top to bottom, assuming you have no prior experience. 

--------------------------------------------------------------------------------
1. WHAT IS THIS PROJECT?
--------------------------------------------------------------------------------
Agromarket is an E-Commerce web application specifically designed for agricultural 
products. It's a marketplace where people can buy and sell items.

Think of it like an Amazon for farmers. It allows users to browse products, 
add them to a shopping cart, apply coupons, and checkout. It also allows 
sellers to list their products, and administrators to oversee everything.

--------------------------------------------------------------------------------
2. THE TECHNOLOGIES (THE TOOLS WE USED)
--------------------------------------------------------------------------------
This project is built using a modern "Tech Stack" (a combination of tools):

* Laravel (PHP): This is the main engine of the app. PHP is a programming language, 
  and Laravel is a framework (a pre-built foundation) that makes building web 
  apps much faster and safer.
* Tailwind CSS: This is used for styling the website to make it look beautiful. 
  Instead of writing complicated custom styles, Tailwind gives us simple words 
  (classes) to color, space, and size elements.
* Alpine.js: A lightweight tool to add small bits of interactivity (like dropdown 
  menus or popups) without writing heavy Javascript code.
* Vite: A super-fast tool that bundles all our CSS and Javascript together so the 
  browser can read it quickly.
* Database (MongoDB/SQL): The place where we permanently save data (users, 
  products, orders). This project is uniquely set up to use either standard SQL 
  databases (like MySQL) or MongoDB (a modern document database).

--------------------------------------------------------------------------------
3. THE ARCHITECTURE: "MVC" EXPLAINED
--------------------------------------------------------------------------------
Laravel uses a pattern called "MVC" (Model-View-Controller). Think of a restaurant:
* The Model is the Chef in the kitchen. They handle the raw ingredients (Data in 
  the database).
* The View is the Plate of food presented to the customer. It's what you actually 
  see on the screen (HTML/CSS).
* The Controller is the Waiter. The waiter takes your order (a web link), goes to 
  the kitchen to get the food from the chef (Model), and brings it back to your 
  table (View).

--------------------------------------------------------------------------------
4. USER ROLES (WHO CAN DO WHAT?)
--------------------------------------------------------------------------------
The app has three main types of users:
1. Farmer (Default User): Can browse products, add to cart, wishlist items, 
   leave reviews, and buy things. They also have an option to apply to "Become 
   a Seller".
2. Seller: Can do everything a Farmer can, plus they have a special dashboard 
   where they can add new products, edit their products, and update the status 
   of customer orders.
3. Admin: The boss. They have an Admin Dashboard where they can manage all users, 
   approve/reject sellers, create product categories, and manage all orders.

--------------------------------------------------------------------------------
5. TOUR OF THE FILES (EVERY NOOK AND CRANNY)
--------------------------------------------------------------------------------
Let's walk through the actual folders in your project and what they do:

📂 app/
The brain of the application. The most important folders here are:
  - Http/Controllers/: (The Waiters). Here you will find files like 
    `CartController.php` (handles adding/removing from cart), 
    `OrderController.php` (handles checkout), and Dashboard controllers for 
    Admins, Sellers, and Farmers.
  - Models/: (The Chefs). Here you will find files representing your data. 
    For example: `User.php`, `Product.php`, `Order.php`, `Cart.php`, `Coupon.php`. 
    There is a special `BaseModel.php` here that magically switches between SQL 
    and MongoDB depending on your settings!

📂 routes/
The traffic cops. 
  - `web.php`: This file holds all the web addresses (URLs). When a user types 
    in "/shop", this file tells the application which Controller (Waiter) to call.

📂 resources/
The face of the application.
  - views/: This contains all the HTML templates (called "Blade" files in Laravel). 
    This is what the user actually sees. You have folders here for `auth` (login/register), 
    `dashboards`, `shop`, and `cart`.

📂 database/
The filing cabinet blueprint.
  - migrations/: These files contain instructions on how to create the database 
    tables. For example, `create_products_table.php` tells the database exactly 
    what information a product needs (name, price, description, etc.).

📂 public/
The front door. This is the only folder the outside world is allowed to see. 
It contains `index.php` (the starting point for every request), and public 
assets like images.

📂 config/
The settings panel. All the configuration files for the database, emails, and 
security live here.

📄 .env (and .env.example)
The secret configuration. This file holds passwords and database connection 
settings. It is never shared on the internet.

📄 composer.json & package.json
The grocery lists. 
  - `composer.json` tells the system what PHP tools (like Laravel itself, or the 
    MongoDB connector) need to be downloaded to make the app work. 
  - `package.json` does the same thing for Javascript tools (like Tailwind and Vite).

--------------------------------------------------------------------------------
6. HOW IT ALL WORKS TOGETHER (AN EXAMPLE)
--------------------------------------------------------------------------------
Let's say a user clicks a product to view it:
1. The user clicks a link: `yoursite.com/shop/product/5`
2. The `routes/web.php` file sees this link and says, "Aha! I need to send this 
   to the `LandingPageController`."
3. The `LandingPageController` acts as the waiter. It asks the `Product` Model 
   (the chef), "Hey, get me the details for product number 5 from the database."
4. The `Product` Model fetches the data and gives it back to the Controller.
5. The Controller takes that data and gives it to a View (like `shop.show.blade.php`).
6. The View formats the data with beautiful Tailwind CSS and sends the final HTML 
   page back to the user's browser.

And that is your entire Agromarket project in a nutshell!
